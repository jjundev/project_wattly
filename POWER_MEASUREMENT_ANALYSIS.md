# Wattly 전력 측정 방식 분석 보고서

- 작성일: 2026-06-23
- 분석 환경: MacBook Pro (Apple M5), macOS 26.5.1
- 비교 대상: Wattly, Mx Power Gadget 1.6.4, 생 `sudo powermetrics`
- 분석 범위: 프로세서 전력, 배터리 충·방전 전력, 표시 평활화, 관련 테스트

## 1. 요약

분석 결과는 다음과 같다.

1. **기존 프로세서 전력은 Mx Power Gadget보다 높게 표시되는 실제 문제가 있었다.**
   - Wattly는 IOReport의 `CPU Energy` 롤업 채널을 CPU 전력으로 사용한다.
   - 이 채널은 개별 CPU 코어뿐 아니라 클러스터 공유 자원(L2 캐시, 인터커넥트 등)의 에너지까지 포함한다.
   - 반면 Mx Power Gadget의 `powermetrics` CPU 값은 실측상 개별 코어 에너지 합에 가깝다.
   - 따라서 특히 유휴·저부하 상태에서 Wattly가 크게 측정됐다.
   - 생 `powermetrics` 게이트로 per-core 범위를 확정했고, 현재 구현은 개별 코어 합을 우선하도록 수정했다.

2. **배터리 방전 전력 계산은 현재 코드와 실기 센서 기준으로 정상이다.**
   - Wattly의 배터리 전력은 SMC `B0AP` 값을 사용한다.
   - 이 값은 별도 SMC 전력 센서인 `PSTR`, `PPBR`, 그리고 `전압 × 전류` 계산과 거의 일치했다.
   - 사용자가 보는 정수 배터리 `%`는 갱신 단위가 크고 느리므로 순간 전력과 짧은 시간 동안 비교하면 전력이 과대해 보일 수 있다.

3. **프로세서 전력과 배터리 전력은 측정 범위가 서로 다르다.**
   - 프로세서 전력: CPU·GPU·ANE 에너지 모델 기반 추정치
   - 배터리 전력: 디스플레이·메모리·SSD·무선·SoC 등을 포함한 전체 시스템 순방전 전력
   - 따라서 배터리 방전 전력이 프로세서 전력보다 큰 것은 정상이다.

## 2. 현재 구현 구조

### 2.1 프로세서 전력

`PowerProvider`는 IOReport 비공개 API의 `Energy Model` 그룹을 구독한다.

- 관련 코드: `Wattly/Providers/PowerProvider.swift`
- 원시 누적 에너지 스냅샷을 두 번 읽는다.
- 이전 값과 현재 값의 차이를 경과 시간으로 나누어 W를 계산한다.
- 30초를 초과한 간격, 카운터 리셋, 엔진 채널 구성 변화, 200 W 초과 값은 폐기한다.

핵심 계산은 `Wattly/Core/PowerEnergy.swift`에 있다.

```swift
case "CPU Energy":                 return .cpu
case "GPU Energy", "GPU":          return .gpu
case "ANE", "ANE Energy", "ANE0":  return .npu
```

```swift
let totalJ = cpuJ + gpuJ + npuJ
```

GPU는 `GPU Energy`를 우선하고 `GPU`를 폴백으로 사용하여 중복 합산을 방지한다. DRAM, DCS, PCIe 등의 채널은 프로세서 카드 합계에서 제외한다.

### 2.2 배터리 전력

`BatteryProvider`의 우선 데이터 소스는 AppleSMC이다.

- `B0AP`: 배터리 순전력(mW, 방전 시 음수)
- `B0AV`: 배터리 전압(mV)
- `B0AC`: 배터리 전류(mA)
- `PDTR`: 어댑터 입력 전력(W)

앱 표시 규칙은 방전을 양수 내부값으로 바꾼 뒤 UI에서 `−` 기호를 붙이는 방식이다.

```swift
func netWatts(batteryMilliwatts mw: Int) -> Double {
    -Double(mw) / 1000.0
}
```

SMC 키를 읽지 못하면 AppleSmartBattery의 `PowerTelemetryData.BatteryPower` 또는 `InstantAmperage`로 폴백한다.

### 2.3 표시 평활화

프로세서와 배터리 카드에는 기본적으로 4초 시정수의 EMA가 적용된다.

```swift
alpha = 1 - exp(-dt / tau)
```

- 원시 측정값은 변경하지 않는다.
- 카드의 숫자와 선택 가능한 평활 그래프에만 적용한다.
- 부하가 갑자기 감소하면 이전의 높은 값이 수 초간 남을 수 있다.
- 이 현상은 순간 비교 편차를 키울 수 있지만, 프로세서 전력 범위 불일치의 근본 원인은 아니다.

## 3. 진단 방법

진단용 CLI 하니스를 임시로 작성하여 Wattly의 실제 프로덕션 소스 파일을 함께 컴파일했다. 하니스는 다음 값을 같은 시간축에서 수집했다.

- `PowerProvider`의 CPU·GPU·ANE·합계
- `BatteryProvider`의 순방전 전력
- SMC `B0AP`, `B0AV`, `B0AC`, `PSTR`, `PDTR`, `PPBR`
- 접근성 API로 읽은 Mx Power Gadget의 PKG·CPU·GPU·ANE 화면값
- AppleSmartBattery 원시 용량

진단 후 하니스와 임시 로그는 모두 삭제했다.

추가로 다음 조건을 분리하여 검사했다.

1. 유휴·일반 변동 부하에서 1초 간격 비교
2. 타임스탬프가 가까운 샘플끼리 정렬한 비교
3. 4개 CPU 코어에 지속 부하를 준 비교
4. IOReport 전체 채널의 에너지 델타 분해
5. 배터리 전력을 세 가지 독립 SMC 값과 비교

## 4. 프로세서 전력 분석

### 4.1 시간 정렬 비교

53개의 시간 정렬 샘플 평균은 다음과 같았다.

| 항목 | Wattly | Mx Power Gadget | 차이 |
|---|---:|---:|---:|
| CPU | 3.796 W | 1.812 W | +1.984 W (+109.5%) |
| GPU | 0.147 W | 0.148 W | −0.001 W |
| 합계/PKG | 3.943 W | 1.941 W | +2.002 W (+103.1%) |

GPU 값이 사실상 일치하므로 다음 가설은 배제할 수 있다.

- 전체 경과 시간 계산 오류
- 모든 IOReport 채널에 공통으로 적용되는 단위 변환 오류
- GPU와 `GPU Energy`의 중복 합산
- Mx Power Gadget과의 단순 타임스탬프 오차만으로 설명되는 편차

편차는 CPU 경로에 집중되어 있다.

### 4.2 IOReport 채널 분해

동일한 측정 구간에서 다음 값이 확인되었다.

| 채널 범위 | 전력 |
|---|---:|
| `CPU Energy` | 3.4086 W |
| `ECPU` + `PCPU` 클러스터 합 | 3.4085 W |
| 개별 E/P 코어 합 | 약 2.509 W |
| 비슷한 시점 Mx CPU | 2.58 W |

`CPU Energy`와 클러스터 합이 오차 범위 내에서 동일하다. 반면 개별 코어 합은 Mx Power Gadget의 CPU 값과 가깝다.

이는 Wattly가 사용하는 `CPU Energy`가 다음을 포함한다는 뜻이다.

- CPU 코어
- 클러스터 공유 캐시
- 클러스터 관리자
- 공유 인터커넥트 등 코어에 직접 귀속되지 않는 CPU 서브시스템 에너지

Mx Power Gadget의 CPU 값은 개별 코어 합에 가까우므로 두 앱은 동일한 이름 아래 서로 다른 범위를 표시하고 있다.

### 4.3 지속 부하 비교

4개 CPU 코어에 지속 부하를 준 상태의 평균은 다음과 같았다.

| 항목 | Wattly | Mx Power Gadget | 차이 |
|---|---:|---:|---:|
| 합계/PKG | 16.785 W | 15.968 W | +0.817 W (+5.1%) |

고부하에서는 개별 코어 전력이 커져 공유 자원의 상대 비중이 감소한다. 그 결과 유휴 상태에서는 2배 이상이던 편차가 고부하에서 약 5%로 줄었다.

이 부하 의존적 편차 역시 고정 배율이나 단순 단위 오류보다 측정 범위 차이라는 결론을 지지한다.

### 4.4 확인된 근본 원인과 원본 powermetrics 게이트

기존 코드에는 다음과 같은 잘못된 전제가 있었다.

> `CPU Energy + GPU Energy + ANE`가 `powermetrics`의 Combined/Package Power와 동일하다.

이 전제는 기존 구현에서 현재 분석 기기에 성립하지 않았다. 추가로 수집한 생 `sudo powermetrics` 30샘플은 다음과 같았다.

| 통계 | CPU Power | Combined Power |
|---|---:|---:|
| 전체 평균(중간 부하 포함) | 3.681 W | 3.898 W |
| 중앙값 | 2.560 W | 2.610 W |
| 유휴 부분집합(CPU < 4 W) 평균 | 2.550 W | — |

유휴 CPU 2.55~2.56 W는 IOReport 개별 코어 합 2.509 W와 가깝고 롤업 약 3.4 W와는 다르다. 이에 따라 CPU는 개별 코어 채널 합을 사용하고, 인식 가능한 코어가 없는 칩에서만 롤업으로 폴백하도록 수정했다.

적용 후 별도 15초 구간에서는 기존 롤업 방식 3.309 W가 per-core 방식 2.664 W로 약 19.5% 감소했다. 같은 구간 Mx GUI 평균은 0.880 W였다. 즉 범위 오류는 교정됐지만, Mx GUI는 깊은 idle에서 생 powermetrics(유휴 약 2.55 W)와 다른 평활·표시 모델을 사용한다. 따라서 정확도 기준은 Mx 화면이 아니라 원본 powermetrics로 둔다.

### 4.5 배제된 원인

다음 문제는 발견되지 않았다.

- 동일 이름의 `CPU Energy` 채널이 여러 번 나타나 중복 합산되는 문제
- `GPU`와 `GPU Energy` 동시 합산
- 현재 기기의 CPU `mJ`, GPU `nJ` 단위 디코딩 오류
- 음수 델타 또는 카운터 리셋으로 인한 스파이크
- DRAM·DCS·PCIe가 프로세서 카드 합계에 포함되는 문제

### 4.6 추가 위험 요소와 조치

현재 문제의 직접 원인은 아니지만 다음 부분도 함께 수정했다.

1. 알 수 없는 단위 라벨을 무조건 `mJ`로 처리하던 동작을 제거했다.
   - 엔진 채널 단위가 nil/미상이면 해당 interval을 드롭하고 리베이스라인한다.

2. sanity ceiling을 500 W에서 200 W로 낮췄다.

3. monitor 폴 시작 시각 대신 IOReport 호출 직전·직후의 중간 시각을 provider 내부에서 기록한다.

## 5. 배터리 방전 전력 분석

### 5.1 독립 센서 비교

59개 유효 샘플의 평균은 다음과 같았다.

| 데이터 소스 | 평균 전력 |
|---|---:|
| Wattly `BatteryProvider` | 15.401 W |
| SMC `PSTR` | 15.443 W |
| SMC `PPBR` | 15.528 W |

차이는 약 0.8% 이내이다. 별도의 짧은 고부하 구간에서도 Wattly 22.853 W와 `PSTR` 22.720 W가 거의 일치했다.

각 샘플에서 `B0AP`는 `B0AV × B0AC`와도 일치했다. 따라서 다음 경로는 정상이다.

- SMC 바이트 순서와 부호 확장
- mW → W 변환
- 방전/충전 방향 변환
- 전압·전류 표시

### 5.2 배터리 `%`와 맞지 않아 보이는 이유

분석 당시 `AppleRawMaxCapacity`는 약 6,257 mAh였다.

- 1% 용량: 약 62.6 mAh
- 15.4 W / 12.3 V: 약 1.25 A
- 이 부하가 유지될 때 1% 감소 예상 시간: 약 3분

따라서 15~25 W가 표시되어도 정수 배터리 퍼센트는 수 분 동안 그대로일 수 있다.

또한 비교 대상이 서로 다르다.

- 카드의 W: 최근 수 초의 순간 또는 EMA 전력
- 배터리 `%`: 배터리 관리 시스템이 평활·보정·양자화한 잔량

짧은 구간의 `%` 변화량을 순간 W로 환산하면 오차가 매우 크다. 배터리 잔량 기반 검증은 최소 수십 분 구간에서 수행해야 한다.

### 5.3 폴백 경로의 한계

SMC를 읽지 못할 때 사용하는 AppleSmartBattery 경로는 주 경로보다 신뢰도가 낮다.

- `BatteryPower`와 `InstantAmperage`의 갱신이 느리다.
- 짧은 구간에서 plateau가 발생할 수 있다.
- 부호가 일시적으로 잘못 보고될 수 있다.
- 현재 코드는 배터리 사용 중에는 절댓값을 방전 방향으로 강제하여 UI 오류를 완화한다.

현재 분석 기기에서는 SMC 주 경로가 정상 작동했으므로 이 한계가 실제 과대 측정의 원인은 아니었다.

## 6. 테스트 커버리지 평가

관련 테스트 33개를 실행했고 모두 통과했다.

- `PowerEnergyTests`
- `BatteryPowerTests`
- `PowerSmoothingTests`

다만 프로세서 전력 테스트는 합성 딕셔너리만 사용한다. 현재 테스트는 다음 규칙만 검증한다.

- 주어진 `CPU Energy` 델타를 W로 올바르게 변환하는가
- GPU 별칭을 중복 합산하지 않는가
- 하위 채널을 합계에 중복 포함하지 않는가
- EMA가 수학적으로 올바르게 동작하는가

즉 `CPU Energy`가 Mx/powermetrics의 CPU 범위와 동일한지는 검증하지 않는다. 테스트가 통과하면서도 실기 의미가 틀릴 수 있는 구조다.

## 7. 개선 권고안

### 우선순위 1: 프로세서 카드의 지표 정의 결정

생 powermetrics 게이트로 선택지 A를 확정했다.

#### 선택지 A — Mx Power Gadget/powermetrics와 일치시키기 (채택)

생 powermetrics 게이트 결과에 따라 채택·구현했다.

- CPU 전력은 개별 CPU 코어 에너지 채널의 합으로 계산한다.
- GPU와 ANE는 현재 방식을 유지한다.
- `_SRAM`, `*DTL*`, 클러스터 롤업 채널은 제외한다.
- 헤드라인은 개별 코어 CPU + GPU + ANE로 구성한다.

주의할 점은 칩 세대별 채널 이름이 다를 수 있다는 것이다. 런타임 채널 탐색과 하드웨어 프로파일이 필요하다.

예상 채널 유형:

- `ECPU0`, `ECPU1`, ...
- `PCPU0`, `PCPU1`, ...
- 이후 칩의 `MCPU*`, `PACC*` 계열

단순 `hasPrefix("ECPU")` 방식은 `ECPU_SRAM`, `ECPUDTL` 등을 잘못 포함할 수 있으므로 정확한 패턴 매칭이 필요하다.

#### 선택지 B — 기존 롤업 범위를 유지하기 (미채택)

- 카드 이름을 `CPU 서브시스템 전력` 또는 `프로세서 클러스터 전력`처럼 변경한다.
- Mx/powermetrics와 동일하다는 주석과 문서를 제거한다.
- 개별 코어 합과 클러스터 공유 전력을 별도 항목으로 분해한다.

현재 UX가 Mx Power Gadget과 직접 비교되는 상황을 고려하면 선택지 A가 더 적합하다.

### 우선순위 2: 실기 차등 검증 하니스 추가

합성 단위 테스트와 별도로 수동 또는 opt-in 하드웨어 검증 도구를 둔다.

권장 수용 기준:

- 30초 이상 구간 평균
- 유휴, CPU 지속 부하, GPU 부하 각각 측정
- Wattly CPU/GPU/합계와 `powermetrics` 비교
- 지속 부하 기준 오차 허용 범위 정의(예: ±5~10%)
- 칩 모델과 macOS 버전을 결과에 기록

### 우선순위 3: 배터리 카드에 장기 평균 제공

현재 순간값은 정확하지만 사용자가 `%`와 비교하기 어렵다.

다음 중 하나 이상을 표시하면 오해를 줄일 수 있다.

- 현재 전력
- 1분 평균
- 5분 또는 15분 평균
- 현재 부하가 지속될 때 예상 잔여시간
- 최근 구간 소비 에너지(Wh)

배터리 잔량과 비교하는 대표값은 4초 EMA보다 최소 1분 평균이 적합하다.

### 우선순위 4: 샘플 타임스탬프 개선

IOReport 스냅샷의 실제 취득 시각을 provider 내부에서 기록한다. 현재처럼 여러 provider가 공유하는 폴 시작 시각을 사용하지 않으면 provider 실행 지연 변화에 따른 오차를 줄일 수 있다.

## 8. 최종 판정

| 항목 | 판정 | 설명 |
|---|---|---|
| 프로세서 에너지 델타 계산 | 정상 | 누적 에너지 차이와 시간 나눗셈은 정상 |
| 프로세서 채널 단위 처리 | 현재 기기에서 정상 | CPU mJ, GPU nJ가 올바르게 변환됨 |
| 프로세서 채널 중복 방지 | 정상 | GPU 별칭과 하위 채널 중복이 방지됨 |
| Mx Power Gadget과의 지표 동등성 | **수정 완료** | 개별 코어 합 우선, 미인식 칩만 롤업 폴백 |
| 배터리 SMC 측정 | 정상 | 독립 센서와 약 1% 이내로 일치 |
| 배터리 부호·단위 계산 | 정상 | 방전/충전 방향 및 mW→W 변환 정상 |
| 배터리 `%` 단기 비교 | 부적절 | 정수 잔량, 게이지 지연, 순간값 차이 |
| 표시 EMA | 정상이나 비교 편차 가능 | 부하 감소 후 높은 값이 수 초간 남음 |
| 실기 회귀 테스트 | 수동 하니스 추가 | `scripts/power-differential.sh`로 sudo powermetrics와 동시 비교 |

핵심 수정 대상은 배터리 계산이 아니라 **프로세서 CPU 채널의 의미와 선택 방식**이었으며, 생 powermetrics를 기준으로 per-core 우선 경로를 적용했다. 배터리에는 `%`와 비교하기 쉬운 τ=60초의 1분 평균도 서브라인에 추가했다.
