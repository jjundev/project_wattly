# Menubar Display Items Update Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add three new selectable menubar-text chips to Settings → 메뉴바 → "표시할 지표" — 배터리 순증감 (W), 메모리 압력 (%), and 자체 소비 전력 (W) — so the menubar text picker catches up with features already shipped in the popover/footer but never exposed as a menubar item.

**Architecture:** `배터리` is a full `CardKind` (`Wattly/Models/CardKind.swift`) that already has a provider, a popover card, and an `@AppStorage` key pattern (`StorageKey.menu(_:)`) — it was just excluded from the menubar-eligible set by an earlier decision. Un-excluding it means inserting it into the existing `CardKind`-keyed pipeline (`MenuBarText.order`/`part`, `Defaults.menuMetrics`, the three `Set<CardKind>` builders). `메모리 압력` and `자체 소비 전력` are NOT `CardKind`s — pressure is a sub-figure of the existing `.mem` card, and self-power has no popover card at all — so they get their own standalone `@AppStorage` flags and are appended to the assembled menubar string as pre-formatted "extra parts", outside the `CardKind` pipeline entirely.

**Tech Stack:** Swift 6 (strict concurrency), SwiftUI, `@AppStorage`/`UserDefaults`, Swift Testing (`import Testing`, `@Test`, `#expect`). Xcode project generated by XcodeGen from `project.yml` (no new files are added in this plan, so `xcodegen generate` is not required).

## Global Constraints

- Swift 6 strict concurrency mode; deployment target macOS 14.0 (per project memory).
- All new UI copy is Korean, matching the existing tone and punctuation style exactly (e.g. `"메모리 (GB)"`, not `"Memory (GB)"`).
- Pure logic (`MenuBarText`, `Accessibility`, `SettingsReset`) MUST be covered by Swift Testing unit tests. SwiftUI Views (`SettingsView`, `MenuBarLabel`, `PollPolicyBridge`) have NO dedicated test files anywhere in this codebase today — that is the established convention here, not an oversight — so Task 4's View wiring is verified by build + manual run, not new unit tests.
- `SettingsReset.applyDefaults` (`Wattly/Core/SettingsReset.swift`) is the single place that writes every persisted `@AppStorage` key back to its `Defaults` value. Every new key added in this plan MUST get a corresponding line there, in the same task that introduces the key.
- `MenuBarText.assemble(selected:states:)` and `Accessibility.menuBarLabel(selected:states:)` are called from tests and other call sites with their current 2-parameter signatures. Any extension MUST be additive via a defaulted parameter — never change or remove an existing parameter.
- Reuse existing formatters instead of duplicating format logic: `CardPresentation.f1(_:)` (one-decimal fixed format) and `CardPresentation.batterySign(netW:charging:)` (the ± sign rule, including the "drop the sign when |net| rounds to 0.0" rule from issue #17).
- Build: `xcodebuild -project Wattly.xcodeproj -scheme Wattly -configuration Debug -destination 'platform=macOS' CODE_SIGNING_ALLOWED=NO build`
- Full test suite: `xcodebuild -project Wattly.xcodeproj -scheme Wattly -destination 'platform=macOS' test`
- Scoped test (faster iteration): add `-only-testing:WattlyTests/<SuiteName>` before `test`, e.g. `-only-testing:WattlyTests/MenuBarTextTests`.

## Decisions

These were confirmed with the user before writing tasks (writing-plans decision checkpoint):

| Decision | Choice | Reason |
|---|---|---|
| Which new metrics get menubar chips | 메모리 압력 %, 자체 소비 전력 W, 배터리 순증감 W | The only scalar metrics that exist in the codebase but have no menubar exposure path today. GPU-temperature cluster breakdown and fan min-RPM were investigated and excluded — both are list/range data suited to the popover expand region, not a one-line menubar string. |
| Memory pressure % vs. the existing memory GB chip | Fully separate, independently-toggleable chip (`메모리 압력 (%)`) | User's explicit choice — lets someone show pressure without GB, or both, or neither. |
| Self-power / battery-net-power architecture | Battery reuses the existing `CardKind`-keyed pipeline (it's already a full card); self-power and memory-pressure get standalone `@AppStorage` flags appended as "extra parts" outside that pipeline | Battery already has a provider/card/storage-key pattern — reusing it is the DRY path. Self-power has no card at all, and memory-pressure must stay independent of the `.mem` GB chip (previous row), so forcing either into the `CardKind` pipeline would be a bigger, unjustified ripple (new fake cards, `CardOrder`/`CardPresentation`/`Thresholds` changes) for a menubar-only figure. |

```swift
    /// The menubar VoiceOver label — "Wattly" plus the selected metrics in canonical order,
    /// computed regardless of whether the visible text is on (issue 15 §1). Decision A: an
    /// empty selection reads just "Wattly". Reuses the #14 assembler so the symbol copy
    /// matches the menubar text exactly.
    ///
    /// `extraParts` (menubar items update) carries the pre-formatted memory-pressure/self-power
    /// figures — they have no `CardKind`, so they can't flow through `selected`/`states`, and
    /// the caller (`MenuBarLabel`) formats them via `MenuBarText.memPressurePart`/`selfPowerPart`
    /// before passing them in. Defaults to `[]` so every existing call site is unaffected.
    static func menuBarLabel(selected: Set<CardKind>, states: [CardKind: MetricState],
                              extraParts: [String] = []) -> String {
        var parts: [String] = []
        if let metrics = MenuBarText.assemble(selected: selected, states: states) { parts.append(metrics) }
        parts.append(contentsOf: extraParts)
        guard !parts.isEmpty else { return "Wattly" }
        return "Wattly, \(parts.joined(separator: "  ·  "))"
    }
```
