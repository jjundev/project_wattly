# Settings Card Unification Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Give every Settings section the same `SettingsCard` chrome (closing the gap where 테마/레이아웃/동작 모드 render bare on the window background and 되돌리기 hand-rolls its own copy of the card chrome), align the Settings card corner radius with the popover's Mode A/B cards, and regroup the 12 Settings sections into three labeled groups (시스템/표시/동작) that reflect what each section actually controls instead of the accreted, plan-by-plan insertion order they're in today.

**Architecture:** Purely a SwiftUI view-layer restyle — no new `@AppStorage` keys, no new persisted state, no change to `SettingsReset`. A new pair of shared geometry constants (`Tokens.cardRadius`, `Tokens.cardPadding`) becomes the single source of truth for what today are independently-chosen literal `12`s (popover Mode A/B cards) and `10`/`12`/`14` (Settings cards) scattered across five files. `SettingsView.body` is restructured from one flat list of 10 section calls into three named groups; a new `SettingsGroupHeader` component labels the two multi-section groups (표시, 동작) — the single-section group (시스템) keeps just its own existing section caption, per the reviewed design's decision #9.

**Tech Stack:** Swift 6 (strict concurrency), SwiftUI, Xcode project generated by XcodeGen from `project.yml` (no files are added or removed in this plan, so `xcodegen generate` is not required).

## Global Constraints

- Swift 6 strict concurrency mode; deployment target macOS 14.0 (per project memory).
- No new `@AppStorage` keys and no persisted-state changes. Settings section order stays hardcoded in `SettingsView.body` (not user-configurable) — `SettingsReset.swift` has no section-order key to reset today and this plan does not add one.
- This plan touches ONLY SwiftUI View code and constant declarations (`Tokens`) — it introduces no new pure functions. There is nothing to cover with Swift Testing. Every task's test cycle is "make the change → build clean" rather than red/green, matching this codebase's established convention for View-only plans (e.g. `2026-07-16-menubar-display-items-update.md` Task 4's SwiftUI-view note; prior plans 19/20/22 in project memory all shipped View-only changes verified by build + manual on-device check, never new test cases for `SettingsView`/popover views).
- `Tokens.cardRadius`/`Tokens.cardPadding` are the ONLY new symbols this plan introduces (Task 1). Every other task is an edit to an existing view.
- Preserve the existing `if monitor.isPresent(.fan) { fanCurveSection }` conditional exactly — the fan curve card must stay hidden on fanless (desktop) Macs after the section reorder (Task 6).
- Popover corner radii that are NOT already `12` (Hero card = 14, panel chrome = 16, both in `PopoverHeroView.swift`/`PopoverContentView.swift`) are intentionally left as bare literals — do not touch them, do not tokenize them. They are a deliberately different visual role from the card radius this plan unifies.
- New UI copy is limited to two group-header labels: "표시" and "동작" (Task 6) — both already used as these groups' Korean names in `SettingsView.swift`'s section-title copy today, so no new terminology is introduced.
- Build: `xcodebuild -project Wattly.xcodeproj -scheme Wattly -configuration Debug -destination 'platform=macOS' CODE_SIGNING_ALLOWED=NO build`
- No files are added or removed by this plan, so `xcodegen generate` is not required.

    /// 표시 그룹: 테마 · 레이아웃 · 표시 지표 · 메뉴바 — 화면에 무엇이 보이는지를 다루는
    /// 섹션들. `시스템` 그룹(아래 `generalSection`, `body`)은 섹션이 하나뿐이라 그룹 헤더를
    /// 붙이지 않고 자신의 `SettingsSection` 캡션만 쓰지만, 이 그룹은 4개라 헤더가 붙는다.

            SettingsGroupHeader(title: "표시")
            themeSection
            layoutSection
            showSection
            menubarSection
        }
    }

    // MARK: 동작 (그룹)

    /// 동작 그룹: 전력 표시(EMA) · 그래프 임곗값 · 팬 커브(조건부) · 동작 모드 · 업데이트 주기.
    /// 팬 커브는 재배치 이전과 동일하게 `monitor.isPresent(.fan)`일 때만 렌더 — 이 가드를
    /// 빠뜨리면 팬 없는 desktop Mac에서 팬 커브 카드가 항상 노출되는 회귀가 생긴다.
    private var behaviorGroup: some View {
        VStack(alignment: .leading, spacing: 18) {
            SettingsGroupHeader(title: "동작")
            smoothingSection
            thresholdSection
            if monitor.isPresent(.fan) { fanCurveSection }
            powerModeSection
            pollSection
        }
    }
